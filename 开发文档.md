# Wrt-Wol 开发文档

版本：`1.0.0`

本文档描述当前有效架构、配置、构建、部署、OpenWrt LuCI、设备管理和后续扩展方式。旧版实现和旧版安装包已放入 `deprecated/`，本文不再保留已废弃的接入方案。

## 1. 项目定位

Wrt-Wol 是一个轻量级 Wake-on-LAN 服务，核心目标是让用户在外部通过机器人或自建页面触发局域网内设备开机。

OpenWrt 端负责：

- 读取设备配置。
- 接收来自飞书、Telegram 或自建中转服务的命令。
- 按设备备注、编号或 `all` 选择目标设备。
- 通过指定网口发送 WoL Magic Packet。
- 在 LuCI 中提供图形化配置页面。

自建服务器端负责：

- 提供远程 Web 页面和 API。
- 保存最近一次开机指令。
- 等待 OpenWrt 端主动轮询并领取指令。

## 2. 当前有效触发方式

### 2.1 飞书机器人

飞书入口使用官方 SDK 的长连接能力。OpenWrt 主动连接飞书服务端，收到消息事件后调用统一命令处理器。

需要配置：

- `feishu.enabled`
- `feishu.app_id`
- `feishu.app_secret`
- 飞书事件订阅选择长连接接收事件。
- 添加 `im.message.receive_v1` 事件。
- 添加机器人收发消息权限。

飞书模式不需要 OpenWrt 暴露公网入口。

### 2.2 Telegram 机器人

Telegram 入口使用 Bot API 长轮询。

需要配置：

- `telegram.enabled`
- `telegram.bot_token`
- 可选：`telegram.allowed_user_ids`
- 可选：`telegram.allowed_chat_ids`
- 可选：`telegram.proxy_url`

国内网络访问 Telegram API 可能需要代理。

### 2.3 自建中转服务

自建模式由两部分组成：

- 云服务器运行 `selfhost.server`，提供 Web 页面和 API。
- OpenWrt 运行 `selfhost.client`，主动轮询云服务器。

这种模式不依赖第三方机器人平台，也不需要 OpenWrt 暴露公网入口。

服务器端配置：

```yaml
selfhost:
  server:
    enabled: true
    host: "0.0.0.0"
    port: 8080
    admin_token: "用于网页/API触发的管理Token"
    device_token: "OpenWrt客户端共享Token"
```

OpenWrt 端配置：

```yaml
selfhost:
  client:
    enabled: true
    server_url: "https://wol.example.com"
    device_token: "OpenWrt客户端共享Token"
    poll_interval: "5s"
```

## 3. 代码结构

```text
cmd/feishu-wol/main.go              程序入口，加载配置并启动所有启用的运行入口
internal/config/config.go           YAML、环境变量、默认值、设备归一化
internal/commands/handler.go        统一命令处理：/list、/on、/last、/whoami
internal/feishu/bot.go              飞书长连接机器人
internal/telegram/bot.go            Telegram 长轮询机器人
internal/selfhost/server.go         自建服务器 Web/API
internal/selfhost/client.go         OpenWrt 自建服务轮询客户端
internal/wol/wol.go                 WoL Magic Packet 构造和发送
openwrt/files/etc/init.d/feishu-wol OpenWrt procd 服务脚本
openwrt/files/etc/config/feishu-wol OpenWrt UCI 默认配置
openwrt/files/usr/lib/lua/luci/...  LuCI 配置页面
scripts/build-ipk.py                OpenWrt/ImmortalWrt .ipk 打包脚本
deploy/selfhost/                    自建服务器 Docker 部署示例
deprecated/                         旧版安装包和旧文档归档
```

## 4. 运行流程

### 4.1 OpenWrt 端

1. `main.go` 调用 `config.Load` 读取配置。
2. 配置通过 `normalizeDevices` 生成完整设备表。
3. 程序创建统一命令处理器 `commands.Handler`。
4. 根据配置启动一个或多个入口：
   - 飞书
   - Telegram
   - 自建服务客户端
   - 自建服务服务器
5. 每个入口收到文本命令后调用 `Handler.Handle`。
6. `/on` 类命令解析目标设备。
7. `main.go` 调用 `sendWoLDevices` 逐台发送 Magic Packet。

### 4.2 自建中转服务

1. 用户访问服务器 Web 页面。
2. 输入管理 Token 和目标设备。
3. 服务器记录一条 wake 指令。
4. OpenWrt 客户端轮询 `/api/poll`。
5. 客户端领取指令并执行 `/on <target>`。
6. 客户端向服务器 `/api/report` 回报结果。

## 5. 配置文件

完整示例见 `config.yaml.example`。

### 5.1 飞书配置

```yaml
feishu:
  enabled: true
  app_id: "cli_xxxxxxxxxxxxxxxxxx"
  app_secret: "xxxxxxxxxxxxxxxxxxxxxxxxxxxxxxxx"
```

### 5.2 Telegram 配置

```yaml
telegram:
  enabled: false
  bot_token: "123456789:your_bot_token"
  allowed_user_ids:
    - "123456789"
  allowed_chat_ids: []
  proxy_url: ""
  drop_pending_updates: true
```

### 5.3 自建服务配置

```yaml
selfhost:
  server:
    enabled: false
    host: "0.0.0.0"
    port: 8080
    admin_token: "change-this-admin-token"
    device_token: "change-this-device-token"
  client:
    enabled: false
    server_url: "https://wol.example.com"
    device_token: "change-this-device-token"
    poll_interval: "5s"
```

### 5.4 WoL 和设备配置

推荐使用 `wol.devices`：

```yaml
wol:
  interface: "br-lan"
  port: 9
  broadcast_ip: "255.255.255.255"
  devices:
    - name: "NAS"
      mac: "AA:BB:CC:DD:EE:FF"
      ip: "192.168.1.10"
      interface: ""
      broadcast_ip: ""
      port: 0
```

字段说明：

- `name`：设备备注，聊天命令可用 `/on NAS`。
- `mac`：目标网卡 MAC 地址。
- `ip`：可选，仅用于展示或后续状态功能。
- `interface`：可选，留空继承全局 `wol.interface`。
- `broadcast_ip`：可选，留空继承全局 `wol.broadcast_ip`。
- `port`：可选，`0` 继承全局 `wol.port`。

兼容旧配置：

```yaml
wol:
  mac_addresses:
    - "AA:BB:CC:DD:EE:FF"
```

当 `devices` 为空时，程序会把 `mac_addresses` 自动转换为：

```text
设备1、设备2、设备3...
```

### 5.5 安全配置

飞书白名单：

```yaml
security:
  allowed_open_ids:
    - "ou_xxxxxxxxxxxxxxxxxxxxxxxx"
  allowed_chat_ids:
    - "oc_xxxxxxxxxxxxxxxxxxxxxxxx"
```

Telegram 白名单在 `telegram.allowed_user_ids` 和 `telegram.allowed_chat_ids` 中配置。

## 6. OpenWrt UCI 配置

LuCI 保存后会写入 `/etc/config/feishu-wol`。

主配置：

```text
config feishu-wol 'main'
  option enabled '1'
  option app_id 'cli_xxx'
  option app_secret 'xxx'
  option interface 'br-lan'
  option broadcast_ip '255.255.255.255'
  option port '9'
  option telegram_enabled '0'
  option selfhost_client_enabled '0'
```

设备配置：

```text
config device
  option name 'NAS'
  option mac 'AA:BB:CC:DD:EE:FF'
  option ip '192.168.1.10'
```

OpenWrt init 脚本会把 UCI 转为环境变量，程序启动时优先使用环境变量。

## 7. LuCI 页面

LuCI 入口：

```text
服务 → Wrt-Wol
```

页面包含：

- 基本设置：启用服务。
- 飞书凭证：App ID、App Secret。
- Telegram：Bot Token、User ID、Chat ID、代理。
- 自建服务：服务器地址、设备 Token、轮询间隔。
- Wake-on-LAN：全局网口、广播地址、端口、兼容 MAC 列表。
- 安全白名单：飞书 Open ID、Chat ID。
- 日志：日志级别、日志文件。
- 设备管理：备注、MAC、IP、单设备网口、广播地址、端口。

服务状态检测使用：

```bash
/etc/init.d/feishu-wol running
```

## 8. 聊天命令

所有入口共用同一套命令：

```text
/list        查看设备列表
/on          唤醒全部设备
/on all      唤醒全部设备
/on 1        唤醒编号为 1 的设备
/on NAS      按设备备注唤醒
/last        查看最近一次唤醒记录
/whoami      查看当前平台用户/会话 ID
/help        查看帮助
```

设备匹配规则：

1. 数字优先匹配设备编号。
2. 完整匹配设备备注或 MAC。
3. 支持备注模糊匹配。
4. 如果模糊匹配到多台设备，会要求用户改用编号。

## 9. 自建服务器部署

```bash
cd deploy/selfhost
cp config.yaml.example config.yaml
vi config.yaml
docker compose up -d --build
```

访问：

```text
http://服务器IP:8080/
```

建议通过 Nginx 或 Caddy 提供 HTTPS。

API 触发：

```bash
curl -X POST https://wol.example.com/api/wake \
  -H "Authorization: Bearer <admin_token>" \
  -d "target=all"
```

`target` 可为：

- `all`
- 设备编号，如 `1`
- 设备备注，如 `NAS`

## 10. 构建

### 10.1 本机编译

```bash
make build
```

手动编译：

```bash
CGO_ENABLED=0 go build -trimpath \
  -ldflags "-s -w -X main.version=1.0.0" \
  -o dist/feishu-wol ./cmd/feishu-wol
```

### 10.2 OpenWrt 包

```bash
VERSION=1.0.0 ARCH=x86_64 BINARY=dist/feishu-wol OUTDIR=dist python3 scripts/build-ipk.py
```

输出：

```text
dist/feishu-wol_1.0.0_x86_64.ipk
```

### 10.3 交叉编译

```bash
make build-mipsel
make build-mips
make build-arm
make build-arm64
make build-x86
```

## 11. OpenWrt 安装

```bash
scp dist/feishu-wol_1.0.0_x86_64.ipk root@<路由器IP>:/tmp/feishu-wol.ipk
ssh root@<路由器IP>
opkg install /tmp/feishu-wol.ipk --force-checksum
rm -f /tmp/luci-indexcache
rm -rf /tmp/luci-modulecache
/etc/init.d/rpcd restart
/etc/init.d/uhttpd restart
```

启动：

```bash
/etc/init.d/feishu-wol enable
/etc/init.d/feishu-wol restart
```

查看状态：

```bash
/etc/init.d/feishu-wol running
ps w | grep '[f]eishu-wol'
logread | grep feishu-wol
cat /tmp/feishu-wol.log
```

## 13. 开发扩展

### 13.1 新增触发入口

新增入口时建议遵循以下模式：

1. 在 `internal/<adapter>/` 新建包。
2. 接收平台消息后提取文本、用户 ID、会话 ID。
3. 调用：

```go
handler.Handle(text, commands.Source{
    Platform: "<adapter>",
    UserID:   userID,
    ChatID:   chatID,
})
```

4. 将返回文本发送回原平台。
5. 在 `cmd/feishu-wol/main.go` 的 `buildRunners` 中注册 runner。
6. 在 `internal/config/config.go` 中增加配置结构、环境变量和校验。
7. 在 LuCI 中补充对应配置项。

### 13.2 新增聊天命令

统一命令在 `internal/commands/handler.go` 中维护。新增命令时只需要修改 `Handler.Handle` 和相关 helper，飞书、Telegram、自建服务会自动复用。

### 13.3 新增设备字段

设备结构在 `config.DeviceConfig`：

```go
type DeviceConfig struct {
    Name        string
    MAC         string
    IP          string
    Interface   string
    BroadcastIP string
    Port        int
}
```

新增字段时需要同步：

- YAML 配置。
- UCI 配置。
- `FEISHU_WOL_DEVICES` 环境变量解析。
- LuCI 设备管理表格。
- 文档示例。

## 14. 排障

### 14.1 LuCI 看不到菜单

```bash
opkg install luci-compat
rm -f /tmp/luci-indexcache
rm -rf /tmp/luci-modulecache
/etc/init.d/rpcd restart
/etc/init.d/uhttpd restart
```

### 14.2 页面显示已停止

```bash
/etc/init.d/feishu-wol running
ubus call service list '{"name":"feishu-wol"}'
logread | grep -i feishu
cat /tmp/feishu-wol.log
```

### 14.3 Telegram 无响应

检查：

- Bot Token 是否正确。
- OpenWrt 是否能访问 `api.telegram.org`。
- 是否需要配置代理。
- 是否误填白名单。

### 14.4 自建服务无响应

检查：

- 服务器端 `admin_token`。
- OpenWrt 端 `device_token` 是否与服务器一致。
- `server_url` 是否能从 OpenWrt 访问。
- 反向代理是否转发了 `/api/poll` 和 `/api/report`。

### 14.5 WoL 包发出但电脑不开机

检查：

- BIOS/UEFI 是否开启 Wake-on-LAN。
- 系统网卡驱动是否开启 WoL。
- 电脑断电后主板是否保留待机供电。
- OpenWrt 发包网口是否为 LAN 桥，如 `br-lan`。
- 广播地址是否适合当前网段。
